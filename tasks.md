### #pageerror-object-serialization — page-error capture renders as `[object Object]`; same root cause spammed as N separate Slack "bugs"
✅ **DONE + LIVE-VALIDATED 2026-07-21.** `core/verification.ts` now exports/uses `pageErrorFromUnknown()`: direct messages stay direct, message-less objects serialize safely (circular/bigint-safe), and empty objects get a descriptive non-placeholder fallback. `AgentBrowser.errorsJson()` now honestly types raw errors as `unknown[]`. `core/slack-bugs.ts` normalizes error signatures (query strings, UUIDs/long ids, stack line/column noise), groups repeats, and adds `N occurrences across M flows` to the existing four-field Bug block. Tests: `src/core/verification.test.ts`, `src/core/slack-bugs.test.ts`; `npm run test:unit` = 19/19 and `npm run build` clean. Live partial exhaustive xp.koyal.ai run (`2026-07-21T17-39-10-412Z`, user-stopped): 227 real page-error occurrences serialized as `{"column":0,"line":0,"text":"InvalidStateError: Transition was aborted because of invalid state","url":null}`, zero `[object Object]`; reconstructing `collectProductBugs()` over its 110 saved steps collapsed 21 failed-step occurrences across 10 flows into ONE report block. Slack/final site-summary did not execute because the user stopped the process with SIGINT, but both downstream surfaces consume this same validated collector.

Found 2026-07-20 night, investigating a live xp.koyal.ai run's Slack post. **Root cause 1 (message loss):** `core/verification.ts:111-114` captures uncaught JS exceptions as `{ message: e.message ?? String(e) }`. The raw entries from `agent-browser errors --json` don't always carry a `.message` field; when one doesn't, `String(e)` on a plain JS object always yields the literal string `"[object Object]"`, discarding whatever real fields were actually present. This corrupts `page-errors.json` at the source (confirmed by reading the raw file — it already contains `{"message":"[object Object]"}`, not just a display-layer bug), so it poisons every downstream consumer: step-summaries, `core/site-summary.ts`'s per-site report, and `core/slack-bugs.ts`'s Slack posts (`errorLines()` in slack-bugs.ts just reads `e.message` and prints it verbatim). **Root cause 2 (over-counting), separate but compounding:** ruled out a stale/never-cleared error buffer as the mechanism (`AgentBrowser.clearSignals()` already calls `errors --clear` before every milestone/probe capture — confirmed via `agent-browser errors --help`); instead, error counts stay **flat within each flow** (e.g. constant 5 across every step of one flow, constant 1 across every step of another) rather than accumulating, which means the *same* uncaught exception is genuinely re-firing on nearly every fresh page load across the site. Reporting each occurrence as its own distinct "product bug" is misleading — a live run posted **17 Slack messages, all `[object Object]`, almost certainly one root cause repeated at 17 checkpoints**, to a real, staff-monitored channel.
**Fix (two parts, NOT yet implemented, awaiting go-ahead):**
1. In `verification.ts`, replace the `String(e)` fallback with a safe serialization (e.g. `JSON.stringify(e)`, falling back to `String(e)` only if that itself throws or yields `'{}'`) so a `.message`-less entry still surfaces its real fields instead of a placeholder.
2. In `core/slack-bugs.ts` / `collectProductBugs()`, dedupe near-identical bugs before posting — same short-title/error-signature within a run → one message with an occurrence count, not N copies.
Ranked 🔴 MUST in the Must-do ranking above per explicit user instruction: doesn't stall a run, but silently produces misleading output on a reporting pipeline already in live production use.
